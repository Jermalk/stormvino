## Architecture understood (KYE)

- Routing fires only when `model="auto"` or `model=""` (ROUTING_TRIGGER_MODELS)
- Stage 1: `_detect_signal()` — has_image→vision, has_tools→web_search, long_context→document, keywords→web_search
- Stage 2: `_route_by_embedding()` — cosine sim against task class centroids (desc + examples)
- Stage 3: assessor LLM — only when profile.use_assessor=True AND embedding score < threshold (0.72)
- `_select_model()`: "fastest" → tier=fast local; "balanced" → last local; "best" → last available
- VLMs in AVAILABLE_VLM_MODELS, NOT AVAILABLE_MODELS → text-path routing must not select them
- Assessor pipe reused for inference when task model == assessor model (qwen3-8b)
- B60: 24,480 MiB total VRAM. Kernel reading reliable. Server estimate (_vram_allocated) overcounts by ~40%.

## Bugs fixed this session

### BUG 1 (fixed af1aaf3): Vision centroid dominated all embedding routing
**Root cause:** Vision examples ("describe this image" etc.) created centroid closer to all short queries than code/general descriptions. Every query scored vision ≥ 0.87.  
**Fix:** `_compute_task_class_centroids()` skips task classes with `signal in {"has_image","has_tools"}`.  
**Result:** Centroids now: document / code / general only. Correct classification verified.

### BUG 2 (fixed a744109): ov_monitor showed wrong KV for assessor
**Root cause:** Monitor used global `kv_cache_size_gb` (6GB) for all models including assessor.  
**Fix:** Server exposes `assessor_kv_cache_size_gb` in /health; monitor uses it for `_assessor` slot.  
**Result:** Assessor row now shows `4.5GB wts  2.0GB KV` correctly.

### BUG 3 (fixed a04cb8b): qwen3-14b OOM on load with assessor resident
**Root cause:** kv_cache_size_gb=6 left insufficient VRAM for qwen3-14b (9.1GB weights) + assessor.  
**Fix:** kv_cache_size_gb 6→4 (26K token context, still ample). Added KV-reduction retry (halve KV) when OOM fires with nothing to evict.  
**Result:** qwen3-14b loads cleanly.

## Open issues

### ISSUE 1: Assessor never fires — threshold too low
**Observation:** Every tested query (including "42") scores ≥ 0.7989. Threshold is 0.72. Assessor is unreachable.  
**Cause:** multilingual-e5-large embeddings in this 3-class space (code/general/document) always produce high cosine similarity; absolute values are not calibrated to 0.72.  
**Options:**
- Raise threshold to 0.85 (most queries use assessor) — expensive, slow
- Raise threshold to ~0.82 (only high-confidence queries bypass assessor)
- Remove assessor path entirely; rely on embedding + rule routing
- Keep as-is: assessor is a safety net, not a primary path

### ISSUE 2: Marginal code/general boundary causes instability
**Observation:** "what should I have for dinner?" scored general=0.799 vs code=0.796 in fast profile (T6 ✓), but code=0.794 in precise profile (P3 ✗). 0.003 gap flips across restarts.  
**Cause:** Centroids recomputed at each startup; float rounding + embedding non-determinism causes ±0.005 variance in marginal cases.  
**Fix options:**
- Add more/better examples to general task class centroid
- Increase general description specificity  
- Add "cooking, food, recipes, personal decisions" examples to general

### ISSUE 3: qwen3-30b very slow (1.3 tok/s, 100s load time)
**Observation:** qwen3-30b-a3b loaded with prefix-cache retry (m_element_type.is_static() error on first attempt). Inference 1.3 tok/s vs 3-5 tok/s for 8b/14b models.  
**Note:** Expected for 30B MoE on single GPU. Acceptable for "laborious" profile use cases.
